 Plan: Wire Multi-Agent Debate into the Simulation Pipeline (v4)                                                                         
 Context

 The multi-agent debate system (multi_agent/) and the simulation pipeline (simulation/) are currently disconnected. Running python
 run_simulation.py --config config/example.yaml only knows about the single_llm agent. The goal is to make python run_simulation.py
 --config config/debate.yaml run the full pipeline end-to-end, with clean 4-layer separation that makes the system PID-ready without
 requiring future refactoring.

 Controller Layer (eval/PID/)

 Mid-debate intervention. Called by multi_agent/runner.py between rounds via a hook.

 ┌────────────────────────┬──────────────────────────────────────────────────────┐
 │         Module         │                         Role                         │
 ├────────────────────────┼──────────────────────────────────────────────────────┤
 │ eval/PID/controller.py │ PIDController.step(rho_bar, js, ov) → PIDStepResult  │
 ├────────────────────────┼──────────────────────────────────────────────────────┤
 │ eval/PID/sycophancy.py │ JS divergence + evidence overlap → sycophancy signal │
 ├────────────────────────┼──────────────────────────────────────────────────────┤
 │ eval/PID/types.py      │ PIDConfig, PIDGains, PIDState, PIDStepResult         │
 └────────────────────────┴──────────────────────────────────────────────────────┘

 The controller operates during the debate (between propose→critique→revise rounds), not after the episode. It reads RoundMetrics from the
  just-completed round, and returns ControllerOutput that the runner applies before the next round.

1. RoundMetrics and ControllerOutput Definitions

 RoundMetrics (what the runner sends to the controller after each round)

 @dataclass
 class RoundMetrics:
     """Metrics extracted from one completed debate round.

     Produced by: multi_agent/runner.py after each propose→critique→revise cycle.
     Consumed by: controller hook (eval/PID/controller.py).
     """
     round_index: int                        # 0-based round number
     proposal_vectors: list[dict]            # per-agent {ticker: size} from proposals
     claim_levels: list[str]                 # Pearl levels (L1/L2/L3) across all claims
     divergence_score: float                 # JS divergence across agent proposals
     consistency_proxy: float                # lightweight consistency heuristic (0-1)
     # consistency_proxy is a cheap local metric (e.g., claim-to-order alignment).
     # The full RCA LLM-based check runs post-hoc. This proxy feeds PID in real-time.

 Where divergence_score comes from: Computed from agent proposals using eval/PID/sycophancy.jensen_shannon_divergence() over the numeric
 proposal vectors (how much agents disagree on position sizing).

 Where consistency_proxy comes from: A lightweight heuristic computed locally — e.g., does each agent's order direction match their stated
  hypothesis direction? No LLM call. This is a cheap stand-in for the full RCA check.

 ControllerOutput (what the controller returns to the runner)

 @dataclass
 class ControllerOutput:
     """Control signal applied by the runner before the next debate round.

     Produced by: controller hook.
     Consumed by: multi_agent/runner.py to modify next-round behavior.
     """
     new_agreeableness: float | None = None    # override agreeableness for next round
     force_extra_round: bool = False           # extend debate by 1 round
     inject_prompt_modifier: str | None = None # extra text appended to agent prompts
     # e.g., "The controller detected potential groupthink.
     #         Challenge assumptions more aggressively this round."

 What PID can modify (the intervention surface):

1. agreeableness — the sycophancy knob already threaded through all prompts
2. force_extra_round — extend the debate beyond max_rounds (up to PIDConfig.T_max)
3. inject_prompt_modifier — free-text nudge prepended to agent system prompts for the next round

 What PID cannot modify: agent roster, model choice, temperature, observation data.

 PID Event Logging

 Each PID intervention is recorded in AgentTrace.pid_events so the trace encodes the full intervention history:

 @dataclass
 class PIDEvent:
     """One PID intervention logged in the trace."""
     round_index: int
     metrics: RoundMetrics        # input to controller
     pid_result: PIDStepResult    # raw PID output (e_t, u_t, beta_new, s_t, etc.)
     output: ControllerOutput     # what was actually applied

 AgentTrace gains: pid_events: list[PIDEvent] | None = None

 This makes traces fully reproducible — you can replay exactly what the controller saw and did.

1. Schema Notes for PID (in_loop mode)

 The existing eval.schema.json v1.2.0 already supports everything needed:


* evaluation_mode: "in_loop" — set when PID was active during the debate
* control_trace: controlStep[] — maps directly from PIDEvent list:
* step_index → pid_event.round_index
* observations → [{signal_id: "consistency_proxy", value: ...}, {signal_id: "divergence", value: ...}]
* actions → [{actuator_id: "agreeableness", payload: {new_value: 0.25}}, ...]
* experiment_config.control.policy.type: "pid" with parameters: {Kp, Ki, Kd, rho_star, gamma_beta}
* run_summary.pid_summary — {retry_count_total, intervention_count_total, instability_index}




future steps


1. Wire PID Controller into the Debate Loop

The PID controller logic is fully implemented in eval/PID/controller.py but not connected to the debate graph. You need an "actuator" that takes the PID output (beta_new) and modifies DebateConfig.agreeableness (or other prompt parameters) between rounds.
Concretely:

* Add a pid_feedback_node to multi_agent/graph.py that runs after revise_node, calls PIDController.step() with the current round's consistency/divergence scores, and updates the config for the next critique-revise loop
* This is what makes H2 (controllability) testable

2. Implement P2-P4 Evaluators

You only have P1 (consistency) in eval/consistency.py. The paper promises P2 (evidential support), P3 (alternative consideration), and P4 (causal alignment / Pearl level). P4 is especially critical since H3 is entirely about regulating it.
Concretely:

* P4 is the easiest to start — you already have PearlLevel tags on every Claim in multi_agent/models.py. Build an evaluator that scores whether claims genuinely demonstrate L2/L3 reasoning (not just labeled as such)
* P2/P3 can follow the same LLM-judge pattern as your ConsistencyJudge

3. Connect Multi-Agent Debate to the Simulation Runner

The simulation in simulation/runner.py uses legacy agents from agents/. It needs to call MultiAgentRunner.run() instead, so debate-generated trades execute against real price trajectories. This is Future Work item 1 in the paper.
Concretely:

* Create an adapter in agents/ that wraps MultiAgentRunner and conforms to the interface expected by AsyncSimulationRunner
* Map case_loader output → Observation format the debate system expects

4. Run the 4-Architecture Comparison

You have MultiAgentRunner (debate) and MajorityVoteRunner (majority vote) already built. You need:

* A single-agent runner (just one agent, no critique/revise) — trivial to create from existing code by setting roster to 1 agent and skipping critique
* Run all four on identical cases with the same observation data
* This directly answers the paper's ablation comparison (Future Work item 3)

5. Run Eval at Scale Across All 8 Ablation Configs

You have 8 configs in multi_agent/demo.py and working consistency + divergence evaluators. The missing step is a batch script that:

* Runs all 8 configs against the same set of cases (live mode, not mock)
* Pipes each trace through ConsistencyJudge and DebateDivergenceAnalyzer
* Collects results into a table for the paper

6. Finish the Data Pipeline

data-pipeline/generate_cases.py is ~60% complete. You need it to produce enough cases for statistically meaningful results across your ablation configs. The impact score formula is implemented but the end-to-end pipeline (fetching → scoring → case JSON output) needs hardening.
