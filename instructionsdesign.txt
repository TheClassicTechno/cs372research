Do You Need LangChain / LangGraph?
Short answer: Not strictly necessary, but LangGraph would help and is worth considering.

Here's the analysis:

Current approach (raw OpenAI SDK)	LangChain/LangGraph
What you have	Direct openai.chat.completions.create() calls, manual JSON parsing, manual orchestration loops	--
Pros of current	Simple, no extra deps, full control, easy to debug	--
What LangGraph adds	State machine for debate flow, built-in retry/fallback, graph visualization, easier to add new rounds/agents as nodes	More abstraction
What LangChain adds	Output parsers (structured output), prompt templates, chain composition	Heavier, more indirection
My recommendation:

LangGraph: YES, worth adopting. Your debate flow (propose -> critique -> revise -> judge) is literally a directed graph with cycles (configurable N rounds). LangGraph models this naturally. It would also make the ablation configs much cleaner (remove a node = remove an agent).
LangChain: OPTIONAL. The structured output parsing (JsonOutputParser) would clean up the manual regex JSON extraction in debate-runner.ts:59-82, but it's not essential.
Key consideration: Your meeting notes say "prefer something easy/debuggable for a class project -- likely combine Claude SDK + LangChain/LangGraph rather than building from scratch." Since Deveen already built from scratch with OpenAI SDK, you could either (a) refactor to LangGraph for your multi-agent orchestrator, or (b) keep extending the current approach. Given this is a research project, I'd suggest keeping the current TypeScript + OpenAI SDK approach but structuring your orchestrator with a graph-like config rather than adding LangGraph as a dependency (which would mean a Python rewrite or adding a separate Python layer).
