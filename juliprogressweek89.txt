PROGRESS REPORT — Weeks 8-9: Multi-Agent Debate → Simulation Adapter
================================================================================

GOAL: Wire the multi-agent debate system (multi_agent/) into the simulation
      pipeline (simulation/) so `python run_simulation.py --config config/debate.yaml`
      runs the full Propose → Critique → Revise → Judge pipeline end-to-end.

--------------------------------------------------------------------------------
1. ADAPTER BUILT (agents/multi_agent_debate.py)
--------------------------------------------------------------------------------
  - Implements AgentSystem ABC (bind_tools / invoke) so the simulation runner
    can use the debate system interchangeably with single_llm
  - Registered as @register("multi_agent_debate") in the agent registry
  - Converts Case → Observation (simulation → debate data format)
  - Converts Action → Decision (debate → simulation data format)
  - Float-to-int order size truncation with validation (skips invalid/zero orders)
  - Side validation (only "buy"/"sell" accepted, case-insensitive)
  - Runs sync MultiAgentRunner.run() via asyncio.to_thread() to avoid blocking
  - Supports mock mode via system_prompt_override: "mock" in YAML config

--------------------------------------------------------------------------------
2. FEATURE ENGINEERING MODULE (simulation/feature_engineering.py)
--------------------------------------------------------------------------------
  - Created build_observation(case) function per integration_plan.md
  - Computes returns from daily bars: (last_close - first_close) / first_close
  - Computes volatility: std-dev of daily log-returns over the bar window
  - Flattens case_data items into text_context with [EARNINGS]/[NEWS]/[INFO] tags
  - Converts portfolio positions from int → float for debate models
  - Lives in simulation/ (Environment Layer) per architectural plan

--------------------------------------------------------------------------------
3. RUNNER MODIFIED (simulation/runner.py)
--------------------------------------------------------------------------------
  - Added broker fallback: when agent returns orders but didn't call the
    submit_decision tool, the runner calls broker.execute_decision() directly
  - This follows integration_plan.md §8: "Agent returns Decision, Runner calls
    broker directly, No tool used"
  - Backward compatible with single_llm (which still uses the tool pattern)

--------------------------------------------------------------------------------
